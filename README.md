# Order Agent Service (ALKE 자동 주문 엔진)
Order Agent Service는 하드웨어 교체 없이 소프트웨어 설치만으로 키오스크에 배리어프리 주문 기능을 제공하는 ALIKE 서비스의 자동 주문 엔진입니다.

<img width="1704" height="916" alt="readme_전체 흐름도" src="https://github.com/user-attachments/assets/2417de24-9ab3-4722-b497-8cd630c3aa00" />


| 기능                  | 설명                                                |
| ------------------- | ------------------------------------------------- |
| 자동 주문 실행            | 사용자가 웹 채팅으로 주문을 완료하면 키오스크에서 자동으로 주문을 수행합니다.       |
| UTG 생성              | 설치 초기 키오스크 UI를 탐색하여 주문 가능한 전체 경로를 그래프로 생성합니다.     |
| UTG 기반 Path Finding | 주문 요청이 들어오면 UTG에서 필요한 클릭 경로를 탐색하여 좌표 순서를 반환합니다.   |
| OCR/LLM 기반 UI 인식    | 초기 UTG 생성 과정에서 버튼, 텍스트, 상호작용 가능한 UI 요소를 추론합니다.    |
| Neo4j 그래프 저장        | UI 전환 관계를 그래프 데이터베이스에 저장하여 경로 탐색과 관리 효율을 높입니다.    |
| 부분 UTG 재생성          | 메뉴나 옵션이 변경된 경우 전체 그래프가 아닌 변경 지점의 하위 노드만 재생성합니다.   |
| 제조사별 UI 대응          | 키오스크 UI 차이를 모듈화하고 전략 패턴 기반으로 처리합니다.               |


## UTG란?
UTG(UI Transition Graph) 는 키오스크의 화면 전환 구조를 그래프로 표현한 데이터입니다. 키오스크 주문 과정은 일반적으로 다음과 같은 흐름을 가집니다.

```
장소 선택 → 카테고리 선택 → 메뉴 선택 → 옵션 선택 → 결제
```

이 흐름을 그래프로 저장합니다.
+ 노드: 특정 화면 상태 또는 선택 가능한 UI 요소
  + 화면 제목
  + 메뉴명 또는 옵션명
  + OCR로 인식된 텍스트
  + 버튼 바운딩 박스
  + 클릭 좌표
  + 주문 단계 정보
  + 하위 노드 관계
+ 엣지(Edge): 특정 좌표를 클릭했을 때 발생하는 화면 전환
  + 클릭 좌표
  + 전환 전 노드
  + 전환 후 노드
  + 클릭 대상 UI 요소
  + 주문 단계
+ 엣지 속성: 클릭해야 하는 좌표, 버튼 정보, 다음 화면 정보
+ 그래프 경로: 하나의 주문을 완료하기 위해 클릭해야 하는 순서
<img width="1134" height="814" alt="readme_utg구조" src="https://github.com/user-attachments/assets/3f01bc59-fc04-45e8-b423-e15cc99f68c6" />


즉, UTG는 “어떤 화면에서 어떤 좌표를 클릭하면 어떤 화면으로 이동하는가”를 저장한 그래프입니다.

이러한 UI 전환 관계는 트리 또는 테이블보다 그래프 구조로 표현하는 것이 자연스럽습니다. 따라서 UTG 저장소로 관계형 데이터베이스가 아닌 Neo4j 를 사용합니다.


## UTG 기반 자동 주문이 필요한 이유
초기에는 주문이 들어올 때마다 OCR과 LLM을 이용해 현재 화면을 분석하고, 필요한 버튼을 추론하는 방식을 고려했습니다. 하지만 이 방식은 다음과 같은 문제가 있습니다.

+ 매 주문마다 OCR/LLM 호출 비용 발생
+ 실시간 추론으로 인한 응답 지연
+ OCR 인식 오류에 따른 주문 실패 가능성
+ 키오스크 UI가 복잡할수록 처리 시간 증가

이를 해결하기 위해 Order Agent Service는 설치 초기 단계에서 키오스크의 모든 주문 경로를 미리 탐색하고, 이를 UTG 형태로 Neo4j에 저장합니다. 이후 실제 주문이 들어오면 OCR/LLM을 다시 호출하지 않고, UTG에서 경로만 탐색하여 자동 주문을 수행합니다.

+ 기존 방식: 주문 요청 → 실시간 OCR/LLM 분석 → 버튼 추론 → 클릭 → 반복
+ ALIKE 방식: 주문 요청 → UTG Path Finding → 좌표 시퀀스 반환 → 자동 클릭

이를 통해 자동 주문의 속도와 안정성을 높이고, 반복적인 OCR/LLM 호출 비용을 줄일 수 있습니다.


## UTG 초기화
서비스 설치 초기에는 관리자 페이지에서 UTG 생성 버튼을 눌러 키오스크의 주문 경로를 생성합니다.

UTG 생성 과정은 점주가 등록한 메뉴와 옵션 정보를 기반으로 진행됩니다. 시스템은 키오스크 화면을 탐색하면서 OCR과 LLM을 활용해 버튼, 텍스트, 상호작용 가능한 영역을 인식하고, 실제 클릭을 반복하며 가능한 주문 경로를 그래프로 저장합니다.

메뉴 수와 키오스크 UI 복잡도에 따라 UTG 생성에는 약 2~10분 정도가 소요될 수 있습니다.

## UTG 수정 및 부분 재생성
키오스크 운영 중에는 메뉴가 추가되거나, 옵션이 변경되거나, UI 배치가 수정될 수 있습니다. 이때 전체 UTG를 처음부터 다시 생성하면 시간이 오래 걸리고 운영 효율이 떨어집니다.

따라서 변경된 지점부터 하위 단계만 다시 탐색하는 부분 UTG 재생성 방식을 사용합니다.
예를 들어 특정 메뉴의 옵션 UI가 변경된 경우, 카테고리나 다른 메뉴의 경로는 그대로 유지하고 해당 메뉴 아래의 옵션 단계만 다시 탐색합니다.

<img width="600" height="421" alt="readme_수정" src="https://github.com/user-attachments/assets/e64606b5-76d1-4aae-b313-ff5ee8a4aedf" />


```
카테고리 변경 → 카테고리 하위 메뉴/옵션/결제 경로 재생성
메뉴 변경     → 해당 메뉴 하위 옵션/결제 경로 재생성
옵션 변경     → 해당 옵션 이후 결제 경로 재생성
좌표 오류     → 해당 노드의 좌표 수정 후 하위 경로 검증
```

이를 통해 전체 그래프를 다시 만들지 않고도 메뉴 변경이나 OCR 오류를 빠르게 보정할 수 있습니다.


## 키오스크 제조사별 UI 대응
키오스크는 제조사마다 UI 구조, 화면 전환 방식, 옵션 선택 방식, 결제 흐름이 다릅니다. 모든 UI 차이를 하나의 거대한 알고리즘 내부 조건문으로 처리하면 유지보수성이 급격히 떨어집니다.

이를 해결하기 위해 ALIKE는 자동 주문 과정을 주요 단계별 모듈로 분리했습니다.

```
초기 진입 단계
메뉴 선택 단계
옵션 선택 단계
결제 단계
```

각 단계는 공통 인터페이스를 따르되, 키오스크 제조사나 UI 유형에 따라 다른 전략을 선택할 수 있도록 설계했습니다.

<img width="1133" height="809" alt="readme_전략패턴" src="https://github.com/user-attachments/assets/8b8a307a-c6e4-4951-9b01-21d50b5fcc1c" />


```
Order Flow
 ├── Initial Step Strategy
 ├── Menu Selection Strategy
 ├── Option Selection Strategy
 └── Payment Strategy
```

이 구조를 통해 새로운 키오스크 UI가 추가되더라도 전체 자동 주문 엔진을 수정하지 않고, 필요한 단계의 전략만 추가하거나 교체할 수 있습니다.
